# Freight dashboard — data architecture (for HTML and portal)

This note captures how the **Everde Freight Dashboard** workbook and **static HTML** must relate so the portal does not show blank or wrong KPIs. It applies to any HTML export under `Everde_Freight_Dashboard*.html` served by the app.

## Where the portal loads HTML

- **API:** `GET /api/freight/dashboard-html`
- **Resolution (newest `mtime` wins among candidates):**
  1. **`FREIGHT_DASHBOARD_HTML`** — if set and readable, that file only.
  2. Otherwise **`public/Everde_Freight_Dashboard*.html`** in the repo (e.g. drop Claude’s export here for local dev before copying to the share).
  3. Otherwise **`PORTAL_DATA_ROOT/Freight/Everde_Freight_Dashboard*.html`** on the share.
- If both (2) and (3) exist, the **newer** file is served so a fresh `public/` copy can beat an older share artifact without editing `.env.local`.
- The route injects a small `<style>` so **`aside.sidebar`** is hidden when viewed inside the portal iframe (outer sidebar replaces tab navigation).
- The HTML is **not** generated by Next.js; it is a **static file** on the share. Missing or stale numbers are fixed by **regenerating that HTML** from the workbook using the rules below, then copying the file into `Freight/`.

## Workbook: hidden backend tabs (source of truth)

The published Excel (e.g. `Everde_Freight_Dashboard_YYYY-MM-DD.xlsx` after the pipeline) includes **hidden sheets** with five-year aggregates. **All metrics for the HTML export should be computed from these sheets**, not from visible dashboard cells that contain `SUMIFS` formulas.

| Sheet | Role |
|--------|------|
| `_history` | Year × Region × Site × Month × Ship Type × Cust Type |
| `_explorer` | Same as `_history` plus Trailer Type |
| `_trailer_history` | Year × Trailer Type × Trailer Size |
| `_3p_explorer` | Third-party only, with Trailer Type |
| `_3p_carrier` | Carrier spend by Year × Region × Carrier |
| `_3p_lane` | Lane detail by Year × Region × Site × Freight ring |
| `_rep_history` | Sales rep performance |
| `_lane_history` | Lane recovery |
| `_cust_history` | Customer history |
| `_sd_history` | Sales director history |
| `_diesel` | Diesel $/gal by month |

### Critical: formula cached values

If Excel was saved **without full recalculation**, formula cells on **visible** tabs often have **`cached=None`** (no stored result). **Do not** rely on reading those formula cells for numbers. Read **values** from the hidden backend sheets (or materialize aggregates in Python from those sheets) when building the embedded JSON for HTML.

## Default filter semantics (for parity with the workbook)

When precomputing numbers for the HTML `dashboard-data` JSON:

- **Current:** Year **2026**, month **YTD (Jan–Apr)**, region **All**
- **Compare:** Year **2025**, month **YTD (Jan–Apr)**

## Metric rules

- **Loads:** `sum(Loads)` from `_history` or `_explorer` after applying the same filters as the UI.
- **Fill %:** `sum(EUs) / sum(Cap)` using the workbook’s intended grain (load-level aggregation as defined in the pipeline — not an unweighted average of per-row ratios unless that matches the sheet logic).
- **$/Mile:** `sum(Cost) / sum(Miles)`.
- **Ratios in general:** Prefer `sum(numerator) / sum(denominator)` over averaging per-row ratios unless a specific view is defined as row-mean.

## Repo tooling

- **`extract_dashboard_html.py`** — Parses the `<script id="dashboard-data">` JSON from a shipped HTML file for QA (tab manifest, optional CSV). It does **not** fix bad source data; it only extracts what is embedded.
- After changing the HTML generator, re-run extraction against a fresh HTML file to verify cells are populated.

## Alternate paths on the share (Claude / JS Files)

- Example HTML: `\\192.168.190.10\Claude Sandbox\JS Files\Freight\Everde_Freight_Dashboard_*.html`
- Example runner: `\\192.168.190.10\Claude Sandbox\JS Files\Freight\_pipeline\update.py` (may include Path-based layout, `cwd=_work`, non-interactive fuel handling, etc.)

**Operational rule:** Whatever tree you treat as canonical for **Run pipeline** (`DataDrops\Freight\_pipeline\` today), copy the **newest generated HTML** into **`DataDrops\Freight\`** so `GET /api/freight/dashboard-html` picks it up. If `JS Files` has a newer HTML generator or `update.py` improvements, merge those changes into the canonical pipeline folder after a successful run rather than maintaining two divergent runners.

## `update.py` publish filename (Windows)

`DataDrops/Freight/_pipeline/update.py` copies `Everde_Freight_Dashboard_v16.xlsx` to  
`Everde Freight Dashboard YTD <M-D-YY> (rebuilt).xlsx` under `Freight/`.  
Use an explicit `f"{month}-{day}-{yy}"` stamp — **not** `strftime("%-m-%-d-%y")`, which raises **`ValueError: Invalid format string`** on Windows.

## After each full pipeline run

1. Confirm the final **`Everde_Freight_Dashboard_v16.xlsx`** (or latest `v*.xlsx`) exists under `_work` and is copied to `Freight/` per `update.py`.
2. Regenerate **`Everde_Freight_Dashboard*.html`** from that workbook using backend-tab reads only.
3. Drop the HTML under **`DataDrops/Freight/`** (canonical), **or** under the repo’s **`public/`** for local dev (same `Everde_Freight_Dashboard*.html` name pattern; newest vs share wins by `mtime`), **or** set `FREIGHT_DASHBOARD_HTML` during testing.
4. Use **Reload view** in the portal; optional: run `extract_dashboard_html.py` on the new file to spot empty `v` fields in the JSON early.
